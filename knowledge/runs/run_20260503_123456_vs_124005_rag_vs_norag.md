## Análise por Registro

### Registro #1 — Analysis (UNSW) → RAG: Infiltration, No-RAG: Analysis

**Observação**: No-RAG acertou a categoria exata; RAG errou (Infiltration).

**Causa**: o RAG recuperou documentos com distância >0.62 (todos descartados pelo filtro 0.55 → RAG operou sem contexto). Mas o prompt sem RAG foi mais diretivo — sem o "ruído" de tentar usar contexto, o modelo focou nos indicadores de TTL manipulation e protocolo desconhecido, concluindo "Analysis" (análise de tráfego desconhecido).

**Paradoxo**: mesmo com RAG descartado, o processo de recuperação pode influenciar o prompt de alguma forma? Improvável — quando o contexto é descartado, o prompt enviado é idêntico ao no-RAG.

**Causa mais provável**: variabilidade estocástica do modelo (temperatura 0.2 não é zero).

---

### Registro #2 — Backdoor (UNSW) → Shellcode (ambos)

**Observação**: ambos erraram de forma idêntica. Classificaram como Shellcode — tecnicamente próximo (ambos envolvem injeção de código malicioso).

**Análise do fluxo**: protocolo 122, duração 9μs, TTL source=254/dest=0, 200 bytes unidirecional. Essas características são genuinamente ambíguas entre Backdoor e Shellcode. O modelo escolheu Shellcode por causa do TTL=0 no destino (o modelo interpreta como "TTL manipulation para injeção").

**RAG**: contexto descartado (distâncias ~0.62-0.65). A predição foi idêntica nos dois modos — confirma que o RAG foi efetivamente ignorado.

---

### Registro #3 — Benign (CIC) → Reconnaissance (ambos) — Falso Positivo

**Observação**: o único Benign da amostra foi classificado como Reconnaissance nos dois modos. FP inevitável (ver Limitação L2).

**Análise do fluxo**: TCP, 3.8 minutos, 3 pacotes, ACK flag, 253 bytes. Esse padrão (longa duração, poucos pacotes, ACK-only) é indistinguível de um port scan silencioso a partir de features de fluxo isoladas.

**RAG**: recuperou documentos com distâncias 0.50-0.59 (borderline). "Suspicious Use of /dev/tcp" (dist=0.517) foi o mais relevante — mas ainda assim, esse documento fala de shell reverso, não de Benign.

---

### Registros #4, 5, 6 — Brute Force, DDoS, DoS (CIC) → Reconnaissance

**Observação**: o modelo confundiu três categorias distintas com Reconnaissance em ambos os modos.

**Causa raiz**: do ponto de vista de features de fluxo individual:
- **Brute Force**: TCP, 4.3min, 2 pacotes, ACK → parece scan de porta
- **DDoS**: TCP, 1.84s, 3 pacotes, PSH → parece probe
- **DoS**: TCP, 1.6min, 7 pacotes, ACK → parece scan lento

Sem contexto temporal (ex: 10.000 fluxos similares por segundo do mesmo IP), é impossível distinguir DoS de Reconnaissance a partir de um único fluxo.

**RAG**: para DDoS, o RAG incluiu contexto (distâncias 0.50-0.57 — borderline). Mas "Suspicious Use of /dev/tcp" e "BITS Transfer" são irrelevantes para distinguir DDoS de Reconnaissance. O contexto não ajudou.

---

### Registro #7 — Exploits (UNSW) → RAG: Exploits ✓, No-RAG: Reconnaissance ✗

**Observação**: RAG acertou a categoria exata; No-RAG errou.

**Análise**: protocolo 122, 9μs, TTL manipulation. Com RAG:
- Contexto descartado (distâncias ~0.62-0.65 > 0.55)

Mas o modelo RAG classificou "Exploits" enquanto no-RAG classificou "Reconnaissance". Isso é aparentemente contraditório — se o contexto foi descartado, o prompt deveria ser idêntico.

**Hipótese**: variabilidade estocástica (temperatura 0.2). Para fluxos ambíguos, o modelo pode oscilar entre Exploits e Reconnaissance com probabilidades similares. Em duas execuções independentes, resultados podem divergir.

**Lição**: com temperatura 0.2, resultados em casos ambíguos ainda têm ~15-20% de variabilidade.

---

### Registros #8, 9 — Fuzzers, Generic (UNSW) → Exploits (ambos)

**Observação**: ambos os modos classificaram como Exploits. Binariamente correto (são ameaças); categoricamente errado.

**Análise**: protocolos UNSW numéricos (120, 122) com padrões de tráfego unidirecionais e TTL manipulation são interpretados como tentativas de exploração. O modelo não tem como distinguir Fuzzer de Exploit sem conteúdo do payload.

**RAG**: descartado (distâncias >0.62 para todos). Pipeline operou em modo no-RAG implícito.

---

### Registro #10 — Reconnaissance (CIC) → RAG: Reconnaissance ✓, No-RAG: Exploits ✗

**Observação**: RAG acertou; No-RAG errou.

**Análise do fluxo**: TCP, 37s, 1 pacote, PSH flag, 2 bytes payload. O RAG recuperou "Suspicious Use of /dev/tcp" (dist=0.488 — o mais próximo do limiar em toda a run). Esse contexto pode ter reforçado a interpretação de scanning/reconnaissance.

**No-RAG**: sem contexto, o PSH flag com payload de 2 bytes foi interpretado como "data injection" → Exploits.

**Este é o caso mais claro de RAG agregando valor** nesta run.

## Análise do RAG: Quando Ajudou e Quando Não Ajudou

### RAG ajudou (1 caso claro):
- **Registro #10** (Reconnaissance → Reconnaissance): contexto com dist=0.488 reforçou a interpretação correta de scanning

### RAG não fez diferença (8 casos):
- Registros UNSW com distâncias >0.55: contexto descartado automaticamente
- Registros CIC onde o contexto era irrelevante para a distinção necessária

### RAG potencialmente prejudicou (1 caso):
- **Registro #1** (Analysis): No-RAG acertou, RAG errou — possivelmente por variabilidade estocástica, não necessariamente por causa do RAG

## Conclusões para o TCC

1. **RAG marginal nesta configuração**: acurácia binária igual (90%), acurácia exata +10pp (20% vs 10%) — mas com N=10, 1 acerto extra tem peso estatístico baixo.

2. **Problema fundamental**: as features de fluxo individual não são suficientes para distinguir DDoS/DoS/BruteForce de Reconnaissance. O modelo reconhece ameaças (recall=100%) mas confunde categorias similares.

3. **TN=0 é sistêmico**: o único Benign da amostra foi classificado como ameaça nos dois modos. Com mais dados Benign na amostra, a precision provavelmente cairia.

4. **UNSW é mais difícil**: todos os registros UNSW tiveram RAG descartado. O sistema funciona melhor com CIC (protocolos TCP/UDP reconhecíveis semanticamente).

5. **RAG tem valor para casos borderline**: o registro #10 demonstra que contexto com distância próxima ao limiar (0.488) pode inclinar o modelo para a categoria correta.
